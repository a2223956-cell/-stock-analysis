# K线数据编造案例分析 (2026-07-15)

## 案例: 通富微电(002156) 79.39数据错误

### 错误链
7/9报告(正确) → 7/14报告(子agent编造) → 主agent context传播 → V1子agent使用 → V1报告(错误)

### 逐层分析

**第1层: 7/9报告(正确)**
- 正确识别79.38为6/25高点(东财API数据)
- 原因: 子agent调用了东财K线API获取数据

**第2层: 7/14报告(错误根源)**
- 将79.39写成"7/10开盘价/涨停价",41处引用
- 实际7/10: 开盘74.00,最高77.67,从未触及79.39
- 原因: 子agent看到79.38(6/25高点),错误归因到7/10,四舍五入为79.39
- 关键: 子agent没有调用K线API验证

**第3层: 主agent context(传播)**
- 读取7/14报告后将错误数据写入delegate_task context

**第4层: V1子agent(使用错误数据)**
- 信任context中的数据,未独立验证

### 根因
子agent编造K线数据(LLM幻觉): 看到上下文中的数字后编造故事,而非调用API验证。

### 解决方案
1. 主agent预拉K线数据写入context(最可靠)
2. 禁止子agent用涨停价/理论值替代真实数据
3. 读取历史报告时校验K线数据(开盘应在最高最低之间)
